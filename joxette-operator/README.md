# joxette-operator

Kubernetes operator for Joxette, built with the [Java Operator SDK](https://javaoperatorsdk.io/)
(JOSDK 5.3.4 + Fabric8 7.7.0) on the Spring Boot 4 starter. Full design:
[`docs/operator-design.md`](../docs/operator-design.md).

## Status — Phase 1 (complete)

Implemented:
- **`JoxetteCluster` CRD** (`joxette.dev/v1alpha1`) — spec mirrors the Helm chart
  values; CRD YAML generated by the Fabric8 crd-generator into
  `target/classes/META-INF/fabric8/joxetteclusters.joxette.dev-v1.yml`.
- **`JoxetteClusterReconciler`** — runs the catalog guardrail, server-side applies
  the desired resources (each owned by the CR for cascading delete), reads
  workload replicas back, and writes `.status` (`Rejected` / `Progressing` /
  `Ready` with desired-vs-ready replica counts). Reschedules every 10s while
  progressing.
- **`CatalogGuardrail`** — rejects unsafe specs (embedded + replicas>1 / split
  scale-out / pekko-management, or a shared backend without a URI), mirroring the
  Helm chart's `joxette.validate` and design doc §5.
- **`JoxetteClusterResources`** — builds the Fabric8 objects: embedded → one
  all-in-one StatefulSet + PVC; shared backend → per-tier Deployments (recorder /
  replay / compaction pinned to 1); a ServiceAccount (wired into every pod) and
  the main Service; plus, in pekko-management mode, a headless Service and the pod
  RBAC (Role + RoleBinding for `pods` read + `coordination.k8s.io/leases` CRUD).

Deliberately out of scope:
- **ConfigMap** — not needed. All non-secret config flows as `JOXETTE_*` env
  (Spring relaxed binding); the pekko HOCON overlay is built in-process by
  `PekkoConfig`. Secrets come via `secretRef`.
- **Admission webhook** — deferred to Phase 3. It needs TLS/cert plumbing
  (cert-manager or a generated CA), and the reconcile-time guardrail
  (`CatalogGuardrail` → `phase=Rejected`, nothing applied) is already a complete
  safety backstop. The webhook only moves rejection earlier (apply-time).

## Status — Phase 2 (complete)

The `RecordedTopic` and `EntityType` reconcilers converge into a running
cluster's REST API (not Kubernetes objects):

- **`RecordedTopic`** → `/topics`. `RecordedTopicReconciler` resolves
  `clusterRef` → Service DNS (`http://<name>.<ns>.svc:8080`), then `TopicConverger`
  diffs desired vs current and POSTs (create) or PUTs (mode/brokerId drift), and is
  a JOSDK `Cleaner` applying `deletionPolicy` (Delete → `DELETE`, Pause → `POST
  …/pause`, Orphan → no-op) on CR removal.
- **`EntityType`** → `/entities`. `EntityConverger` creates/updates the type
  (buckets drift) and upserts each declared source mapping (server-side
  `POST …/sources` is idempotent); `deletionPolicy` is Delete | Orphan.
- **`JoxetteRestClient`** — thin JDK-`HttpClient` wrapper over the endpoints used;
  **`ClusterRefResolver`** builds the Service DNS; **`RestClientFactory`** is the
  injection seam tests substitute.

Convergers are tested against a real in-process `HttpServer` emulating the API
(create / unchanged / drift-update / delete / pause / orphan paths).

## CRDs

Three kinds in `joxette.dev/v1alpha1`, all generated under
`target/classes/META-INF/fabric8/`:
`joxetteclusters`, `recordedtopics`, `entitytypes`.

## Remaining (Phase 3)

Admission webhook (TLS/cert), ServiceMonitor/HPA wiring, generated OpenAPI client,
and `/health`-driven `clusterRef`-not-ready rescheduling for the API reconcilers.

## Build & run

```bash
# unit tests (guardrail, resource builder, context load)
mvn -pl joxette-operator test

# generate the CRD YAML
mvn -pl joxette-operator process-classes
#  -> target/classes/META-INF/fabric8/joxetteclusters.joxette.dev-v1.yml

# operator image (buildpacks)
mvn -pl joxette-operator spring-boot:build-image
```
